# Техническое задание (исходный текст)

Источник: передано пользователем в чат (PDF "Задачник_new"). Ниже — рабочая текстовая версия для разработки.

## Назначение системы

Система электронного документооборота для автоматизации процесса обработки документов от создания/получения до отправки корреспонденту или завершения исполнения и списания в дело.

## Основные функции

- регистрация входящих, исходящих и внутренних документов;
- учет резолюций, выданных по документам руководством организации, и постановка документов на контроль;
- контроль исполнения документов в рамках подразделений;
- списание документов в дело;
- ведение информационно-справочной работы;
- формирование делопроизводственных отчетов.

## Сущности и обязательные поля

Поля со `*` — обязательные.

### 1) Сотрудники (executors)

- `<id_executor>`
- `*<login>`
- `<fio>`
- `*<unit>`
- `<passwd>` (по умолчанию `Qwerty123`)

Требование: смена пароля через web; принудительная смена раз в 90 дней.

### 2) Записи (inbox)

- `*<inc_out_text>` Исх(внешн)
- `<inc_out_date>`
- `*<id_employer>` → БД employers
- `*<inc_reg_text>` Вх.
- `*<inc_reg_date>` Вх.дата (по умолчанию: сегодня)
- `<about1>` Тема входящего
- `*<expiration>` Выполнить до (по умолчанию `+30` дней от создания; продление без изменения `<inc_reg_date>` не предусмотрено)
- `<id_executor>` → БД executors
- `*<out_reg_text>` Исх.
- `*<out_reg_date>` Исх.дата
- `<out_sender>` Кому направлен (по умолчанию = `<id_employer>`)
- `<about2>` Тема исходящего (по умолчанию = `<about1>`)
- `<broadcast>` Передан в рабочем порядке (по умолчанию пусто)
- `*<status>` строго `{1,0}` = `ВЫПОЛНЕНО/НЕ ВЫПОЛНЕНО`

### 3) Контрагенты (employers)

- `*<id_employer>`
- `*<employer_full>` Полное имя
- `<employer>` Сокращенное имя
- `*<employer_addr_law>` Юр. адрес
- `<employer_addr>` Факт. адрес

## Роли и доступ

Роли: `root`, `user`.

- Без авторизации: только просмотр текущих задач.
- Пользователь: создание, редактирование, удаление своей записи, изменение статуса (согласно ограничениям).
- Root: работа с удаленными записями (удалить полностью, восстановить, переназначить пользователя).

## Интерфейс

Вкладки:
- АКТУАЛЬНЫЕ (по умолчанию)
- ВЫПОЛНЕННЫЕ

Базовые поля в списке:
`<inc_num><about1><expiration><executor.fio><out_out_date><out_sender>`

## Отображение, сортировка, поиск

- По умолчанию показываются актуальные (`НЕ ВЫПОЛНЕНО`) по подразделению пользователя.
- Сортировка: по сроку выполнения + приоритет контроля.
- Если «НА КОНТРОЛЕ»: жирный шрифт и выше в списке.
- Поиск под пользователем по полям:
  - `<inc_out_text>`, `<inc_reg_text>`, `<expiration>`, `<login>`, `<out_reg_text>`
- Поиск выполняется по всем задачам: актуальным, архивным, удаленным.

## Цветовая индикация сроков

- зеленый — от 7 до 4 дней до срока;
- желтый — от 3 до 1 дня;
- красный — текущий день и просрочка;
- иначе — без заливки.

## Статусы, архив, редактирование

- При `ВЫПОЛНЕНО` запись уходит в архив.
- Архивные записи доступны для чтения; статус можно переключить обратно на `НЕ ВЫПОЛНЕНО`.
- Полноценный доступ к полям — у создателя записи.
- Часть полей может заполняться другими исполнителями (исходящие/дата/кому отправлен/и т.п.).
- Поле статуса изменяет только создатель записи.

## Удаление и восстановление

- Удаление делает только владелец записи.
- При удалении владелец меняется на `root` + фиксируется дата удаления.
- Полное удаление — только root.
- Восстановление — только root, с назначением пользователя.
- Множественное удаление запрещено.

## Аудит изменений

Последние изменения фиксируются в отдельном нередактируемом поле:
- дата
- время
- пользователь

Показывается в диалоге редактирования только для чтения.

## Рабочий процесс (как описан в ТЗ)

- Пользователь видит краткую информацию актуальных задач и цветовые метки сроков.
- Для просмотра/изменения полной информации — авторизация.
- Запись состоит из полей с разными правами изменения (владелец/root/все сотрудники для полей закрытия).
- Любой работник может создать запись и становится владельцем.
- Архивные записи возвращаются в актуальные при смене статуса обратно.
